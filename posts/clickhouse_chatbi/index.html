阿里云发布 Quick BI，聊聊 ChatBI 的底层架构、交互设计和云计算生态

Aug 30, 2025 in AGENT
olap chatbi
2 min read

8月28日，阿里发布新一代数据分析工具 Quick BI。它整合了超级数据分析师“智能小Q”和云计算基础设施，实现了问数、归因、生成报告等多种功能。

智能小Q其实是一个多智能体系统（multi-agent system），内含多种 Agent：

  • 报告 Agent
  • 问数 Agent
  • 搭建 Agent
  • 解读 Agent

每个 Agent 均由领域大模型驱动。领域大模型参数量小、速度快，而且在特定领域表现更优。这些模型通过 SFT, RL 训练，积累业务知识与各种细节，再将知识迁移、泛化到同类问题中，带来效果提升。

一、功能演示

我们来看看小Q能做什么？它有两种界面：问数界面和解读界面。

1）问数界面

用户提问：店日均杯量是多少？

小Q可以查询数据并可视化（如下），这就是“问数”。

2）解读界面

解读界面是数据面板，小Q在侧边栏中。用户可以自由地筛选数据面板中的数据，并基于筛选后数据进行提问。

问数界面的劣势在于，Agent 可能无法精确定位到你希望分析的数据表。当你实际知道你要问哪张表的时候，最好的方式就是告诉大模型，你想分析哪张表，甚至告诉大模型具体的数据范围。这正是解读界面在做的事。

Agent 框架虽然缓解了上下文限制，但若你想把有限的上下文资源都用来分析问题，下钻到更深的层次，最好还是给大模型足够明确的上下文信息。

在与智能体的对话中，明确地引用所指对象是好文明！

二、拆解 Quick BI 技术路径

Quick BI 的底层架构是 Agent + 数据库。只要有这两味材料，你也能做出简易版 ChatBI，这一点在我的开源项目 luochang212/clickhouse-chatbi 中亦有记载。在这两味材料中，Agent 又是重中之重，是本次技术革命的创新点。我们不禁要问 Quick BI 的 Agent 如何实现？

先来盘一盘 Agent 有哪些能力？

当前的 Agent 能力，大而不强。无论是 TRAE，还是 Quick BI，只要是基于 Agent 框架开发的产品，都难以跳出现阶段 Agent 科技的限制。

1）当前阶段的科技限制

什么限制？我举几个例子你感受下：

  1. 长期记忆：如何精炼有用对话，遗忘无用对话，跨对话提升问答效果？
  2. 验证能力：光有记忆没用，还要学会判断，也就是 Verification 的能力
  3. 知识体系：有验证能力之后，如何运用验证能力将记忆加工成知识？

这些就是 Agent 下一步需要努力的地方。因为涉及工程、模型的复杂性，彻底解决这些问题还需要时间。

那么，基于现有 Agent 能力，我们能开发出怎样的 ChatBI 呢？

问得好！ChatBI 能力中，最简单的就是问数了。

2）最简单的功能：问数

以实现问数功能为例，看看 Quick BI 需要做到哪些事：

  1. 找到数据表
  2. 获取表注释、字段注释
  3. 写 SQL，取数
  4. 分析数据，得出结论
  5. 以报告或图表的形式呈现

以我的亲身经验，问数在技术上很简单，使用 ReActChat 模式即可轻松完成上述任务。难的东西不在 Agent，而在数据生态建设。难的是表注释是否清晰、字段注释是否完整、是否包含对枚举值的解释这种小事。

良好的数据生态建设、完善的业务知识库是问数功能迈向好用的第一步。

在生产环境中，数据多，数据表可以达到上千张；生产环境还很“脏”，无论数据流程有多精密，口径对不齐的情况总会发生。这是因为数据表是人开发的，而对人的管理又是有极限的。数据规模一大，就乱象从生，亟须治理。

这就显得工程师文化的重要了。表的命名要一致、清晰；字段注释要完整；业务细节要写在 wiki 中。好的公司、好的文化、好的人 才能创造出相对干净的生产环境。没有这些基础条件，后边的数据智能化之路只会愈发艰难。

4）Agent 如何连接数据库

注：MAS 是多 Agent 系统的意思

Agent 主要通过 MCP 连接数据库。比如在 ReActChat 模式，Agent 会采取走一步看一步的策略，每次通过 MCP 查询数据库获取的增量信息，都会作为下一步行动的决策依据。逐步推理，最终逼近答案。

除了 MCP，Agent 还能通过工作流或分析引擎连接数据库。在复杂业务中，使用自带 RAG 的工作流是一种常见做法。RAG 可以检索业务知识库，将业务知识融入上下文，或者利用业务知识做 Query 改写，从而提高回答质量。

三、云+AI的战略

听完 Quick BI 发布会，我理解阿里是不得不做这项业务的。因为 Quick BI 是云计算业务的外延，它能延续阿里在云时代的优势。

云计算和 Quick BI 耦合得有多紧密？

可以说，没有好的云计算基础设施，就无法发挥 Quick BI 的全部实力。云基础设施落后的结果是：数据部分上云，表和字段注释不清，没有SLA，也没有即席查询的能力。以前没有这些还能马马虎虎过日子，现在都变成了必须回头补课的东西… 不解决这些基础层面的东西，Quick BI 轻则不准，重则不 work。所以呀，希望将 BI 升级到 ChatBI 的公司，恐怕不得不购买配套的云基础设施。

而且，云服务还呈现极强的生态粘性。在有选择的情况下，拥有阿里云基础设施的公司，最好选择 Quick BI 作为 ChatBI 解决方案；而部署了 Quick BI 的公司，也最好选择阿里的云基础设施，否则需要承担高昂的兼容成本。不出意外，Quick BI 的发布将带动一波阿里云基础设施的销售。

因为 AI，云计算的门槛更高了？

云计算本就是“属于”大厂的业务，因为它技术复杂，小厂容易玩不转。云计算技术有多复杂？报个菜名，让大家感受一下：OLTP、OLAP、ETL、Ad-hoc、Hadoop、Spark、Flink、Airflow、Dubbo、Docker、K8s、Kubeflow… 这些都是云计算的相关概念，但不过是冰山一角罢了。

随着 AI 技术的加入，云计算市场向头部厂商集中的趋势正在增强。据 澎湃新闻 报道：吴泳铭认为，相比传统的云计算，AI 所产生的变革，将极大提高云计算市场的集中度。对于这件事，Quick BI 本身就是一个例证。无论是产品丰富度，还是技术复杂度，大厂都更有优势赢得这场战争。
