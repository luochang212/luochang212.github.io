Hadoop 本身是用 Java 写的，所以使用 Java 编写 MapReduce 框架是最原生的方法。但随着 Hadoop 影响力扩大，其他语言的程序员也希望使用自己的编程语言编写 MapReduce。于是 Hadoop Streaming 应运而生，它用 Unix pipeline 在任务间传递数据，使得我们几乎可以使用任何一个主流语言编写 MapReduce。

PS: 本文主要关注 Python 在 Hadoop Streaming 上的应用。

过去半个多世纪，随着算力的高速增长，半导体工业似乎达到了它的物理极限，摩尔定律正在失效，寄托于更好更快的芯片来提高算力的期望正在落空。但业务的增长却不会因此而中止，人们亟需一个解决方案来满足对我们算力的渴求。

分布式计算似乎是个好主意。“当古人用一头牛拉不动一根原木时，他们从未想过培育更高更壮的牛。同样，我们也不需要尝试开发超级计算机，而应该试着结合使用更多的计算机系统。” (Haddop 权威指南)。但分布式计算的发展存在进程间交互、数据存储、分布式算法等瓶颈。Google 的几位工程师为了解决这些痛点，开发了 Hadoop。

Hadoop 是一个分布式计算框架，它建立在计算机集群上，负责协同集群上的机器共同工作。Hadoop 处理了分布式计算中的大部分难题，大量复杂细节都无需用户考虑，用户只需要告诉 hadoop 应该如何处理数据就可以了。

Hadoop 将数据处理抽象成 MapReduce 框架。顾名思义，MapReduce 包含了两个过程，Map 和 Reduce。Hadoop 编程，其实主要就是编写 MapReduce （以及一系列烦人的配置文件）。

Note: 对 Hadoop 原理感兴趣的同学，可以看看这个 -> MapReduce: Simplified Data Processing on Large Clusters

1. MapReduce

OK，终于进入正题了。

MapReduce 的重要性不必详说。MapReduce 描述了数据在 Hadoop 中是如何被处理的。数据处理可大致分为三个过程：Map, Shuffle 和 Reduce。其中，Shuffle 由 Hadoop 完成，因此我们编写 Map 和 Reduce 就好。

1.1 数据处理三阶段

下图展示了数据在 Map, Shuffle, Reduce 三阶段中的流向。

三阶段的作用：

  • Map：预处理。Shuffle 通常是要耗费大量的计算资源，因此传给 Shuffle 的数据越精简越好，这就要求我们在 Mapper 阶段把和研究问题无关的数据都筛掉。Mapper 在输出时，会指定一个或多个字段作为主键。

  • Shuffle：排序 (merge) + 分桶。Shuffle 首先对 Mapper 传来的数据按主键排序，再将有序数据分桶。分桶指的是将已排序数据切割，然后分发给 reducer 的过程。 （我有个疑问：主键相同的数据是否一定会被分入同一个桶中？懂的大神请告我一声）

  • Reduce: 规约数据。将大数据规约为我们关心的问题。可能是一个或一组统计量，也可能是一组经过规约的原始数据。

1.2 kv 表

数据在 map, shuffle 和 reduce 三个阶段中，是通过 kv表 的形式在每个阶段间交换的。kv 表以一个或多个字段作为 key（主键），其余作为 value。在 Unix 中 kv 表一般就长这样子👇。下面的每行数据，各字段间是以制表符 \t 作为分隔符的。 

dogs    1    happy
cats    3    happy
pigs    2    sleeping
cats    1    sad

1.3 一个例子

下面举例说明数据在 Hadoop 中是如何被处理的。

我们用 Hadoop 来执行 WordCount 任务。WordCount 的要求很简单，就是统计文本中每个单词出现的次数。

用非 Hadoop 的方法处理该问题只需要几行 Python 代码👇。

# word_count.py
# usage: cat test_file.txt | python word_count.py
import sys
import collections


words_dict = collections.defaultdict(int)

for line in sys.stdin:
    line = line.strip().split(' ')
    
    if not line:
        continue
    
    for word in line:
        words_dict[word] += 1

for k, v in words_dict.items():
    print('{}: {}'.format(k, v))

但在大数据场景下，这种代码的单机运行时间会非常长，甚至可以长至几天，因此就有必要学习 Hadoop 实现啦。Hadoop 实现的代码会比下述代码复杂几倍，但这也可以理解，毕竟 Hadoop 实现的代码需要套用 MapReduce 编程模型。世间没有免费的午餐，虽然运行时间降低，但我们写 code 的难度却上升了。

☆ 数据在 Hadoop 中是如何被处理的

在开始写 MapReduce 代码前，先简单描述一下数据在 Hadoop 中的处理过程，这有助于我们后续理解代码。

  1. 原始数据长这样

    i love you
you love him
he loves her
she loves him

  2. 在 Hadoop Streaming 中，数据首先被切分为多个 part (split 操作)

    # part_1
i love you
you love him
✄ -----------------
# part_2
he loves her
she loves him

  3. 每个 part 送入各自的 mapper 做预处理，输出以单词为主键的 kv 表

    # part_1
i    1
love    1
you    1
you    1
love    1
him    1
✄ -----------------
# part_2
he    1
loves    1
her    1
she    1
loves    1
him    1

  4. shuffle 阶段，按主键排序并分桶

    # part_1
he    1
her    1
him    1
him    1
i    1
✄ -----------------
# part_2
love    1
love    1
loves    1
loves    1
she    1
you    1
you    1

  5. reduce 阶段，规约数据

    # part-00000
he    1
her    1
him    2
i    1
✄ -----------------
# part-00001
love    2
loves    2
she    1
you    2

最终我们会在输出目录下得到两个文件：part-00000 和 part-00001。文件数量和 reducer 的数量相同，本例中有两个 reducer 所以输出两个文件。

2. 用 Python 写 MapReduce

利用 Hadoop Streaming 我们可以用 Python 来写 MapReduce。

要知道 Hadoop 本身是由 Java 写成的，Hadoop Streaming 利用标准输入输出 (stdin / stdout) 桥接了 MapReduce 编程模型中的各个阶段，使得 Hadoop Streaming 几乎能支持所有后端语言。但也同时因为需要利用标准输入输出来传递数据，因此运行效率相比原生 Hadoop 来讲要慢一些。

以下代码摘自 Writing An Hadoop MapReduce Program In Python，无意冒犯，侵删。因为他的注释写得太完备了，所以还是决定贴他的版本。非常推荐阅读原文哦，他写得比我好多了

mapper

#!/usr/bin/env python
"""mapper.py"""

import sys

# input comes from STDIN (standard input)
for line in sys.stdin:
    # remove leading and trailing whitespace
    line = line.strip()
    # split the line into words
    words = line.split()
    # increase counters
    for word in words:
        # write the results to STDOUT (standard output);
        # what we output here will be the input for the
        # Reduce step, i.e. the input for reducer.py
        #
        # tab-delimited; the trivial word count is 1
        print '%s\t%s' % (word, 1)

reducer

#!/usr/bin/env python
"""reducer.py"""

from operator import itemgetter
import sys

current_word = None
current_count = 0
word = None

# input comes from STDIN
for line in sys.stdin:
    # remove leading and trailing whitespace
    line = line.strip()

    # parse the input we got from mapper.py
    word, count = line.split('\t', 1)

    # convert count (currently a string) to int
    try:
        count = int(count)
    except ValueError:
        # count was not a number, so silently
        # ignore/discard this line
        continue

    # this IF-switch only works because Hadoop sorts map output
    # by key (here: word) before it is passed to the reducer
    if current_word == word:
        current_count += count
    else:
        if current_word:
            # write result to STDOUT
            print '%s\t%s' % (current_word, current_count)
        current_count = count
        current_word = word

# do not forget to output the last word if needed!
if current_word == word:
    print '%s\t%s' % (current_word, current_count)

3. 测试

一个 Hadoop 任务要跑很长时间，如果你的代码因为有 bug 终止任务，你就亏大了。因为重跑可能要花很长时间，一个粗心大意可能你今天就没法准时下班了。所以在跑 Hadoop 之前，测试是必要的。

我一般会按下面这个流程检查代码：

  1. code review
  2. local test
  3. check try except

Code review 就不细说了，这不是今天的主题；

Local test 是指构造数据文件，然后用 cat data_file | python mapper.py | sort | python reducer.py 命令在本地运行，测试 MapReduce 代码中是否存在语法错误；

Check try except 指检查是否写了充分的 try except 语句来处理可能出现的问题，比如字段缺失、数据类型异常、数值异常等。因为代码需要经受住大数据量的考验，因此对健壮性的要求很高。

4. 文件操作

Hadoop 通过 HDFS (Hadoop Distributed File System) 来管理集群上的文件。

常用命令：

# 查看帮助文档
hadoop fs -help
# 查看当前文件夹的文件
hadoop fs -ls
# 下载文件到本地
hadoop fs -copyToLocal [hdfs路径] [本地路径]
# 上传文件到HDFS
hadoop fs -copyFromLocal [本地路径] [hdfs路径]
# 新建HDFS目录
hadoop fs -mkdir [hdfs路径]
# 删除HDFS文件
hadoop fs -rm [hdfs路径]

总之和 Unix 系统下的命令差不多。

5. 漫谈

Hadoop MapReduce 是一套编程模型，基本思路是先把数据切成不同分片，然后基于键值对排序并在机器间交换一次数据，接着按主键对数据分桶，最后规约到最终结果。我们发现这套模式是死的，也就是你只能按照既定的运算框架和运算顺序来处理数据。

由于模式是死的，因此存在一部分问题天然不适合用 Hadoop 解决。事实上具体到每个问题，适用 Hadoop 的程度都是不一样的。对于有些本来不太适合但又不得不用的，我们还是需要拆分需求/设计运算流程，尽量地去贴 MapReduce 这套模型。需求拆得对不对，流程设计得好不好，在很大程度上会影响 MR 的运行效率。
